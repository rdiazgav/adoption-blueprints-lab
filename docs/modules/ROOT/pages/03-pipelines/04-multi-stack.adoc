= Multi-Stack Pipelines: Python and Node.js
:navtitle: Multi-stack pipelines

Estimated time: *20 minutes*

[.lead]
RetailFlow is a polyglot application: four services are Quarkus/Java, one is Python/FastAPI, and one is Node.js/React.
This scenario runs the Python `recommendations` service and the Node.js `frontend` service through the `generic-service-pipeline` â€” a simpler pipeline that skips Maven and delegates all build tooling to the service's own Dockerfile.

== Two Pipelines, Not One

It is tempting to build a single universal pipeline with conditionals:
"if language == java, run Maven; else skip it."
Tekton supports this with `when` expressions on Task entries, but the result is hard to read and harder to debug.

A cleaner approach is two pipelines with different shapes:

* `quarkus-service-pipeline` â€” clone â†’ Maven build+test â†’ Buildah â†’ Quay â†’ deploy
* `generic-service-pipeline` â€” clone â†’ Buildah â†’ Quay â†’ deploy

Each pipeline does exactly what its service stack requires, nothing more.
Adding a third language (for example, Go) means adding a third pipeline â€” not extending an existing one with more conditionals.

== Side-by-Side Comparison

[cols="3,3,3",options="header"]
|===
| Step | `quarkus-service-pipeline` | `generic-service-pipeline`

| Clone source
| `git-clone` Task
| `git-clone` Task

| Language-specific build
| `maven-build` Task (`./mvnw package`, tests, JAR assembly)
| â€” (handled inside Dockerfile)

| Container image build
| `buildah` Task, Dockerfile: `./src/main/docker/Dockerfile.jvm`
| `buildah` Task, Dockerfile: `./Dockerfile`

| Push to registry
| `buildah` Task (push step)
| `buildah` Task (push step)

| Deploy to OpenShift
| `deploy-to-openshift` Task
| `deploy-to-openshift` Task

| Workspaces
| `source`, `maven-cache`, `docker-credentials`
| `source`, `docker-credentials`

| Quality gate
| Maven Surefire (JUnit) â€” blocks on test failure
| Dockerfile `RUN` steps â€” blocks on non-zero exit
|===

== Run the Frontend Pipeline

The `frontend` service uses Node.js and React.
There is no pre-built example PipelineRun for frontend â€” create one inline:

[source,bash]
----
oc create -f - <<EOF
apiVersion: tekton.dev/v1
kind: PipelineRun
metadata:
  generateName: frontend-run-
  namespace: retailflow
spec:
  pipelineRef:
    name: generic-service-pipeline
  serviceAccountName: pipeline-sa
  timeouts:
    pipeline: 30m
  workspaces:
    - name: source
      persistentVolumeClaim:
        claimName: pipeline-source-pvc
    - name: docker-credentials
      secret:
        secretName: quay-credentials
  params:
    - name: GIT_URL
      value: "https://github.com/rdiazgav/adoption-blueprints-lab.git"
    - name: GIT_REVISION
      value: "master"
    - name: SERVICE_NAME
      value: "frontend"
    - name: SERVICE_DIR
      value: "apps/frontend"
    - name: IMAGE_TAG
      value: "latest"
EOF
----

Watch the pipeline in real time:

[source,bash]
----
tkn pipelinerun logs -f --last -n retailflow
----

== Verify All Services Built

After running pipelines for `catalog`, `recommendations`, and `frontend`, list all PipelineRuns:

[source,bash]
----
tkn pipelinerun list -n retailflow
----

.Expected output
[source]
----
NAME                              STARTED         DURATION   STATUS
frontend-run-x9k2m                3 minutes ago   2m30s      Succeeded
recommendations-pipeline-run-...  8 minutes ago   3m15s      Succeeded
catalog-pipeline-run-v7jq4        15 minutes ago  9m42s      Succeeded
----

Confirm all three Deployments were updated:

[source,bash]
----
for svc in catalog recommendations frontend; do
  echo -n "$svc: "
  oc get deployment "$svc" -n retailflow \
    -o jsonpath='{.spec.template.spec.containers[0].image}'
  echo
done
----

.Expected output
[source]
----
catalog: quay.io/rdiazgav/retailflow-catalog:latest
recommendations: quay.io/rdiazgav/retailflow-recommendations:latest
frontend: quay.io/rdiazgav/retailflow-frontend:latest
----
